# 智件：从“复用软件”到“复用思路”的范式改变

> 文轩注：虽然这一部分文档是AI写的，但实际上这是我构思satis系统的核心。来自于我在课题组讨论时，有人提到“LLM”可以“当做硬件”。但实际上，LLM是软件，对吧！那么我在想的是，“软件”本身，也可以当做“硬件”一样，是可重复使用的。那新的一层是什么呢？是思路。目前的harness engineering在我的角度来看，还是**没有将执行流和思路流解耦**。
>
> 这段时间，OpenClaw、CodeX、Claude Code等工具陆续问世。在使用这些工具的时候，我突然感觉到SKILLS设计的巧妙之处。SKILLS就是一种比较原始的smartware，但它的约束实在是太松了，导致LLM很容易造成Token过度浪费的情况。我想统一所有软件的接口，让软件开发者为satis预留一个接口，这样satis上若接一个Agent框架（目前还没做，我觉得我一己之力肯定比不过大厂的集体力量。不过我总认为这个好办），就可以合法、简单地调用软件了。而软件具体怎么实现，留给软件开发者吧！相信软件开发者能实现足够好的软件，而Satis只需要像苏格拉底一样，发挥脑内的聪明才智，通过简单的接口来驾驭这些软件。
> 
> Satis不是一个Agentic System，但实际上是可以行使Agentic System的功能的。比如Deep Scientific Research这种任务，用一个Agentic System来做，我想效果是逊色于用satis做的。因为Research往往就涉及到多种工具调用。未来工具越来越多，如果每种工具都需要大量的Agentic RL或者后训练来规范，那么所需的显卡卡时得爆炸了。推理的时候，万一某一个工具使用上错误，又要花大量时间改工具，Token要爆炸了。但是，如果将软件的使用抽象出来，看做某种“工具”，并对这个工具进行足够好的抽象，让它统一接口并易于使用，这样说不定就能在尽量少的控制流内，解决尽量多的问题了呢！
>
> 反观satis，若未来有一种LLM能够快速掌握智件的使用，那么软件工程师和智件工程师是分开的。软件工程师专注于制造符合Satis接口的软件，而智件工程师专注于怎么快速准确查询工具用法、并结合实际情况调用工具，这样控制流和思路流是分开的，这个问题就变得好办多了。

过去几十年，软件工程的默认前提是：  
**软件是人类与硬件之间的接口，也是主要的复用单元。**

这个前提没有错，但它隐含了一个限制：  
我们复用了“能做什么”（功能），却很难复用“为什么这么做、在什么条件下这么做、失败后如何改道”（思路）。

`satis` 试图改变的，正是这一层。  
在这里，软件回归为稳定计算单元，而可复用的核心上升为**思路单元**，我把它叫做：**智件（smartware）**。

---

## 1) 为什么说这是范式转换，而不只是新术语

传统软件复用强调的是“函数、模块、服务”的重用，本质是**操作重用**。  
但实际复杂任务中，真正稀缺的常常不是“一个 API 怎么调”，而是：

- 先做哪一步，后做哪一步；
- 出现分歧时按什么依据分支；
- 何时探索，何时收敛；
- 如何把经验沉淀成下一次可复用路径。

所以我在系统里做了一个关键分离：

- **软件（software）**：固定、可调用、可校验的计算能力；
- **智件（smartware）**：围绕目标组织步骤、分支、约束与反馈的可复用思路。

换句话说，软件解决“能不能算”，智件解决“怎么算才对”。

---

## 2) 在 `satis` 里，智件并非抽象概念，而是有机制载体

这套思路不是停留在文档口号，而是已经落到实现机制：

- `Chunk + Plan` 结构把“局部执行”与“全局编排”拆开；
- `task / decision / planning` 使流程能在确定执行与动态分支之间切换；
- `handoff -> edges/depends_on/entry_chunks` 把依赖关系从自然语言提升为可计算结构；
- `render/validate/submit/run` 形成“草稿可探索、运行要收敛”的工程闭环。

因此，智件不是一段 prompt，也不是一个脚本片段。  
它是**可编辑、可验证、可运行、可复盘**的思路结构体。

---

## 3) 软件在新范式中的角色：从“主角”回到“可替换部件”

现在代码里的 software 机制已经体现了这个变化：

- 通过 `SoftwareRegistry` 做统一注册与索引；
- 用 `forsatis.json` 声明函数与参数约束；
- 用 `SKILL.md` frontmatter（`name/description`）声明语义与用途；
- `Software refresh` 递归扫描并重建 `SKILLS.md` 索引，保证注册表与文档同步；
- 运行时通过 `SoftwareCallStmt` 按规范调用 `python_script` / `binary` runner。

这意味着：  
软件能力是标准化、可替换的“算子层”；智件则在其上组织目标导向的“决策层”。  
当某个软件替换版本时，只要接口契约还在，智件仍可复用；当任务目标变化时，智件可改编，而不必重写全部软件。

---

## 4) “思路复用”到底复用了什么

智件复用的不是代码行数，而是四种更高价值的资产：

- **目标展开方式**：把含糊目标拆成可执行阶段；
- **判定逻辑**：什么时候走 A，什么时候走 B（`decision` 分支）；
- **收敛策略**：什么时候允许草稿，什么时候必须强校验；
- **失败处置经验**：失败后回退、重试、改道的路径模式。

这些资产在传统体系里常散落在口头经验、注释和临时聊天里。  
而在 `satis` 里，它们被固定进 plan 图、handoff 关系、运行事件与审计记录中，能够跨人、跨任务复用。

> Satis这一套的目的本来是给AI for Science的Deep Research的，但实际上，Agent系统很难做到思路和实现的解耦。我的论据可能不够充分，但实际上我觉得至少对于我这个领域，是比较合理的。

---

## 5) 智件与软件的边界：避免“全都智能化”的幻觉

这套设计有意保持克制，不把一切都叫“智能”：

- 软件负责**稳定执行**，追求确定性、可测试性、参数契约清晰；
- 智件负责**策略组织**，允许试探、分支与上下文驱动调整；
- 两者通过结构化接口连接，而不是相互侵入。

这条边界非常重要。  
如果把思路硬编码进软件，系统会失去机动；  
如果把软件能力全交给“临场 improvisation”，系统会失去可靠性。

---

## 6) 一个更实用的结论：未来可迁移的是“智件库”

在这个范式下，组织真正可复利的资产，不再只是软件仓库，而是：

- 一套可持续演进的智件模板；
- 一组可审计的决策路径；
- 一批已验证的“目标 -> 路线 -> 执行 -> 反馈”闭环。

软件仍然必要，但它更像“标准化工具箱”；  
智件才是“任务智能的复用单元”。

从工程视角看，这就是我希望 `satis` 推进的变化：  
**把复用重心从“功能组件”上移到“思路组件”，让系统真正积累的是解决问题的方法。**
