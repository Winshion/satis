# chunks 和 plans：如何编辑，以及为什么这样设计

`Chunk` 和 `Plan` 的编辑，不是一个纯 UI 问题，而是一个“如何把人类工作过程结构化”的问题。  
如果一句话概括：**Chunk 负责“这一步怎么做”，Plan 负责“下一步该走哪条路”**。

系统内维护三种规划层级：意图（Intent）、计划（Plan）和执行单元（Chunk）。意图一旦定下便不允许更改，且赋予其整个系统下唯一的ID；Plan在Intent下，是具体的执行单元连接而成的拓扑。而每一个执行单元由若干执行语句（Satis Internal Language, SatisIL）组成。每一个Chunk和每一个Plan在同一个Intent内部share同一个ID。

> 文轩注：从工程上，当前的Plan、Chunk和Intent管理系统以及Workbench TUI写得稀烂，因为都是「走一步看一步」迭代出来 + Vibe Coding出来的成果。UI上，我还是更相信人类的审美以及特调，LLM无法理解我的全部需求。但由于目前目标只是迅速迭代。并验证我的想法的可行性，所以也没有怎么管。未来在UI和管理 Plan/Intent/Chunk 上，会做更多的重新规划。

## 1) 先把“编辑对象”分层：局部可确定，全局可生长

> 文轩注：AI写的，我审核了基本没问题，同时修改增添了一些补充。

系统把编辑拆成两层：

- **Chunk 层（局部）**：编辑单个节点的 `satis_text`、输入输出与 handoff；
- **Plan 层（全局）**：编辑节点之间的依赖边、分支、入口与跨 plan 串联关系。

这背后的工程判断是：
人类在一个具体动作内，执行路径通常是稳定的；但多步任务整体上，方向会随信息变化而调整。在大的战略规划上，“走一步看一步”、“因时而动”是较好的策略；不应轻易定下规划，但一旦定好路线，就要坚决执行。

因此我设计系统的时候，不把“局部动作”与“全局路线”混成一个大脚本，而是分开建模，降低复杂度耦合。在Chunk层面，路径是确定的，系统按照严格的DAG格式来调度。而一个Chunk规划完成后，系统可以选择性终止Chunk（规划结束）、继续接下一个Plan（继续按照已有路径规划）、或者重新运行。

## 2) Chunk 怎么编辑：把“单步动作”做成可控原子

> 文轩注：AI写的，我审核了基本没问题，这部分是专门写给开发者的

在 Workbench 里，Chunk 的编辑核心是三件事：

- 改正文：直接编辑 `satis_text`（这一段就是当前步骤的可执行内容）；
- 改输入：通过 `handoff_inputs` 声明它吃谁的输出（`from_step` + `from_port` + `var_name`）；
- 改类型：在 `task` 和 `decision` 间切换（`Ctrl+T`），保持 `chunk_id` 不变，但语义切换。

同时，系统给了“硬边界”来保证单步编辑不失控：

- 只能删叶子 chunk，不能删根，也不能删最后一个节点；
- 只填一半 handoff 会报错；
- `decision` 的分支用结构化命令维护（`branch`、`decisiondefault` 等），避免手改 JSON 造成漂移。

这其实是在把“一个步骤、一个步骤做”的直觉变成机器可验证的约束：  
用户可以自由改每一步，但每一步都必须是**定义完整、输入清晰、语义明确**的原子操作。

## 3) Plan 怎么编辑：把“走一步看一步”变成显式增量

Plan 的编辑不是一次性定稿，而是“可增长图”的持续修订。系统里有三种典型增量动作：

- **图内增量**：新增子 chunk、补边、改分支，让已有图继续长；
- **片段增量**：`attachfragment` 把新节点/新边追加进旧图，不替换历史；
- **跨 plan 增量**：`plan-continue` / `plan-change` / `plan-detach` 在父子 plan 之间挂接、替换、剥离。

这组能力对应的是“先走到可见位置，再决定下一段路”，而不是“先把全地图算完再出发”。  
技术上，系统通过 `Next Plan` / `Return Plan` 这样的导航节点，把“未来再决定”的空间内置进图结构。

从机制上看，“战略确定、战术灵活”靠的是同一个闭环：

- 战略层先锚定：`intent_id`、`goal`、`plan_id` 这类全局语义约束，保证“为何而战”不漂移；
- 战术层可机动：`plan-change`、`attachfragment`、`decision branch` 允许局部改道，但改道是显式、可追踪的；
- 收敛层做裁决：`render/validate/submit/run` 把临时机动压回统一约束，避免系统长期处于“游击状态”。

> 这里让AI补充了一些《孙子兵法》的隐喻。

## 4) 为什么先允许草稿，再要求收敛：编辑态和运行态分离

> 文轩注：AI写的，我审核了基本没问题，这部分后续如果有更好的UI，可以删掉重新设计。由于UI不是主要矛盾，所以这里先临时做了一个Workbench TUI，仅供参考。另外，我觉得WebUI挺好的，浏览器很优美，而且操作灵活、渲染很漂亮，后续可能往这方面走。

系统故意允许 Workbench 保存 invalid draft，但在 `render` / `submit` / `run` 时强制校验。  
这不是放松质量，而是把两个阶段分开：

- **探索阶段**：允许临时不完美，降低试错成本；
- **执行阶段**：强约束收敛（如单入口、结构合法、依赖可解），保障可运行性。

对应到人的工作方式，就是：

- 想法阶段：可以粗糙；
- 行动阶段：必须严谨。

所以这里的“走一步看一步”不是随意，而是**先探索、后收敛**的工程化流程。

## 5) 关键机制：用 handoff 把“关系”从文本里抽出来

> 文轩注：这也是AI写的，我审核了。改了一些，主要是给开发者看的。

很多系统把依赖藏在自然语言里，最后只能靠猜。  
这里选择把依赖显式化：由 handoff 推导 `edges`、`depends_on`、`entry_chunks`。

好处是三层的：

- **可追踪**：你知道每个输入来自哪个步骤的哪个端口；
- **可重算**：改了 handoff 可以自动重建拓扑；
- **可诊断**：依赖错了会直接暴露为结构问题，而不是运行后隐性失败。

换句话说，系统把“我感觉这一步依赖上一步”升级为“我明确声明并可验证这条依赖”。

## 6) 两个看似矛盾的出发点，如何落到同一套哲学里

> 文轩注：AI写的，有删改

你提到两点：

1. 人做事在具体细节上，是“一个步骤、一个步骤”，且步骤内部路径基本确定；  
2. 人做事在实际做法上，又是“走一步看一步”。
3. 枢念系统在概念上，将做事情的规划思路，和具体的执行步骤解耦，所以1和2中所说的都要兼收并蓄。

这套 Chunk/Plan 设计给出的统一解法是：

- **在 Chunk 内追求确定性**：把一步做成可执行、可复现、可替换的原子；
- **在 Plan 上容纳不确定性**：把路线变化做成图的增量编辑与显式分支；
- **在运行前统一收敛**：通过校验、规范化与事件观测，把探索结果压缩成可执行事实。

如果换成更直接的机制描述，可以理解为两级控制：

- **一级控制**：入口一致性、依赖可解、结构合法性，是“不可谈判”的红线；
- **二级控制**：节点增删改、分支重定向、跨 plan 挂接，是“在红线内”的自由度。

总的来说，Chunks和plans的设计以局部确定性托住全局不确定性，再通过持续收敛把不确定性变成下一步的确定性。

## 7) 对开发者的实践建议（简洁版）

- 先把每个 Chunk 当“最小可交付动作”，再谈整体编排；
- 依赖尽量走 handoff，不要把关键关系藏在正文描述里；
- 大改路线时优先用 fragment / plan-continue 这类增量方式，避免整图重写；
- 每轮改动后都走一遍 `render -> run -> events/inspect`，让结构与现实持续对齐。

当你按这个节奏工作时，Plan 不再是静态流程图，而是一个可演进的“工作推理记录”。

> 文轩注：而且这个推理记录是可审计、可动态扩展的。在我看来挺美的：是一种「大方向保持不变，最小行动单元严格按控制流进行，和战术上“走一步看一步”」三位一体的统一。
